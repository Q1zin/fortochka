# Форточка - открой завесу к своим местам

## Концепция

Сервис «окно к своим местам»: пользователь ставит старый iPhone или Android-телефон с камерой в любимое место (вид из офиса, дача, дом бабушки, аквариум), а на других телефонах (iOS и Android) этот вид автоматически появляется как регулярно обновляемые обои. Ключевая ценность — не «красивые виды вообще», а **свои** места: обои превращают телефон в форточку в другое место.

## Роли устройств

1. **Телефон-камера** (iOS или Android): закреплён стационарно, периодически (каждые N минут) снимает кадр и загружает его на сервер.
2. **Сервер**: принимает снапшоты, хранит последний кадр по каждой камере, обрабатывает (кроп под экран получателя, коррекция экспозиции, опционально ватермарк времени) и отдаёт по стабильному HTTPS-URL, возвращающему чистый JPEG (не HTML-страницу).
3. **Телефоны-зрители** (iOS и Android): периодически подтягивают свежий кадр и ставят его как обои.

## Критичные платформенные ограничения (проверены)

### iOS (зрители)
- **Живые обои с видео из стороннего приложения невозможны** — iOS не даёт приложениям динамически рисовать обои. Единственный механизм — приложение «Команды» (Shortcuts): действия «Получить содержимое URL» → «Установить обои», плюс автоматизации по времени (с iOS 16.4 срабатывают без подтверждения пользователя).
- Интервального триггера в автоматизациях нет — нужно создавать несколько триггеров по времени (например, каждые 30 минут = 48 автоматизаций либо разумный набор из 8–16).
- Обои на устройстве должны быть в режиме «Фото», а не «Photo Shuffle» — иначе Set Wallpaper не сработает.
- URL должен возвращать непосредственно данные изображения.
- Продуктовое следствие: для iOS-зрителя приложение генерирует/устанавливает готовый шорткат (или даёт ссылку на его установку с персональным URL камеры) и ведёт пользователя по настройке автоматизаций. Существует рабочий прецедент: публичные шорткаты, которые берут кадр веб-камеры по URL, кропают под разрешение экрана с центрированием на выбранной точке и ставят как обои.

### iOS (камера)
- Фоновая периодическая съёмка на iOS ограничена: приложение должно быть на переднем плане (сценарий «телефон на подставке, всегда включён, приложение открыто, guided access») — это приемлемо для стационарной камеры на зарядке.

### Android (зрители)
- Два уровня:
  1. **Простой (MVP)**: периодическое обновление статичных обоев — `WallpaperManager.setBitmap()` из фоновой задачи (WorkManager, минимальный интервал 15 минут). Надёжно, почти не ест батарею.
  2. **Честные живые обои**: собственный `WallpaperService` — рендер в `Surface` из `Engine`; можно показывать статичный кадр и подгружать свежий, а в перспективе — живое видео (ExoPlayer/HLS) только когда обои видимы (`onVisibilityChanged`), с паузой на мобильной сети.
- Прецедент-ориентир: Muzei (опенсорсный движок живых обоев с плагинной архитектурой) + плагин Muzei Webcam, показывающий изображение по URL как обои. Оба открыты, можно изучить подходы.

### Android (камера)
- Полноценный фоновый режим: foreground service с CameraX, съёмка по таймеру, работа при выключенном экране, автозапуск после ребута, отключение battery optimizations (нужен онбординг-гайд для пользователя).

## Архитектура (MVP)

```
[Телефон-камера] --(JPEG upload, HTTPS, N мин)--> [Сервер]
                                                     |
                                   хранение последнего кадра per camera
                                   кроп 9:19.5 / под разрешение зрителя
                                   обработка (экспозиция, время)
                                                     |
[iOS-зритель]  <--(GET /cam/{token}/wallpaper.jpg)---+
  Shortcuts: Get Contents of URL -> Set Wallpaper    |
[Android-зритель] <----------------------------------+
  WorkManager -> download -> WallpaperManager.setBitmap
```

- **Пейринг**: камера при регистрации получает ID; зрители подключаются по короткому коду / QR / ссылке-приглашению. Один аккаунт (или семейная группа) — несколько камер и несколько зрителей.
- **Снапшот-URL**: стабильный per-camera per-viewer URL с токеном в пути (т.к. Shortcuts удобнее работать с простым GET без заголовков). Токен отзываемый.
- **Приватность**: кадры — приватные по умолчанию, доступ только по токену; сервер не публикует и не индексирует. Опция: полностью локальный режим (зритель тянет кадр напрямую с телефона-камеры в той же сети / через VPN типа Tailscale) — пост-MVP.
- Сервер также может принимать кадры не только с телефонов, но и с любых IP-камер (RTSP → ffmpeg-снапшот `ffmpeg -rtsp_transport tcp -i rtsp://... -frames:v 1 -q:v 2 out.jpg`) — расширение аудитории почти бесплатно.

## Скоуп MVP

1. Android-приложение «камера»: foreground service, съёмка раз в 15 мин, upload.
2. Сервер: приём кадров, хранение последнего, выдача JPEG по токен-URL, кроп под портретный экран.
3. Android-приложение «зритель»: подключение по коду, WorkManager-обновление обоев (домашний и/или локскрин).
4. iOS-«зритель» без нативного приложения: веб-онбординг, который выдаёт персональный URL и готовый шорткат + инструкцию по автоматизациям.
5. Онбординг-гайды: закрепление телефона, отключение оптимизаций батареи, режим «Фото» для обоев iOS.

Пост-MVP: iOS-приложение камеры, живые обои Android (WallpaperService, затем HLS-видео), несколько камер с ротацией, «золотые часы» (чаще обновлять на рассвете/закате), поддержка IP-камер, локальный P2P-режим.

## Риски и открытые вопросы

- **iOS UX**: настройка автоматизаций Shortcuts — самое трудное место воронки; нужен максимально разжёванный онбординг (видео, deep-links в Команды). Riск: Apple может менять поведение Set Wallpaper между версиями iOS.
- **Живучесть телефона-камеры**: вендорские киллеры фоновых процессов на Android (Xiaomi/Huawei и т.п.) — нужны инструкции per-vendor и watchdog (push/alarm для рестарта сервиса).
- **Нагрев/деградация батареи** телефона на постоянной зарядке — рекомендации по лимиту заряда, интервалу съёмки.
- **Приватность/юридическое**: камера может захватывать людей; нужны дисклеймеры и приватность по умолчанию.

## Первые шаги разработки

1. Сервер: минимальный API (upload кадра, GET последнего кадра по токену, пейринг-коды) + кроп.
2. Android-камера (CameraX + foreground service + upload).
3. Android-зритель (WorkManager + WallpaperManager).
4. Шорткат-шаблон для iOS + веб-страница онбординга, генерирующая персональную ссылку на шорткат.
